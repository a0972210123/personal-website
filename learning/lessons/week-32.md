## 1. 課文

### 1.1 Build vs Buy 第一關：core 還是 context

第一性原理直接繼承 W16：**絕不外包核心能力的載體，盡量外購非差異化的一切**。Geoffrey Moore 的語言：

- **Core**：顧客選你（而非對手）的理由所在——做到世界級會贏得顧客的活動
- **Context**：必要但不加分——做到世界級顧客也不會多付一毛錢（薪資系統再頂尖，客戶無感）

判別問句：「**這項活動若做到世界第一，顧客會更愛我們嗎？**」會calendar→core，自建並持續投資；不會→context，買現成的、用標準品、能外包就外包。會計系統、郵件伺服器、CRM 對多數公司是 context；推薦演算法對 Netflix 是 core、對一家書店的官網是 context——**core/context 由你的定位（W14）決定，不由技術的酷炫程度決定**。

常見病：工程師想自建一切（有趣＋履歷），採購想外購一切（便宜＋省事）——兩邊都在用自己的偏好代替策略判準。

### 1.2 Build vs Buy 第二關：TCO（總持有成本）

過了 core/context 關才算錢，而錢要算全：

- **買**的全成本：授權費＋導入客製＋**供應商鎖定**（lock-in：遷出成本就是你未來的議價力損失——W15 供應商力：對方知道你走不掉，續約年年漲價）＋供應商倒閉/停止維護的風險
- **建**的全成本：開發只是頭期款——**維運人力常是建置成本的 3–5 倍**（整個生命週期）＋佔用稀缺工程資源的**機會成本**（W3：這批工程師若去做 core 功能值多少？）＋自建系統的單點知識風險（會的那個人離職呢）

經驗法則：context 系統的自建提案，把維運與機會成本算進去後，九成不成立；core 系統的外購提案，把 lock-in 與差異化流失算進去後，多半也不成立——**兩關合起來，答案通常自己浮現**。

### 1.3 技術債：Ward Cunningham 的絕妙比喻

1992 年 Cunningham 在 OOPSLA 報告中發明這個比喻：**為了快速出貨而選的「湊合方案」是一筆借款**——本金是未來把它改正的重工，**利息是此後每一次開發都因為要繞過這坨湊合而變慢**。

比喻的精妙在於它繼承了財務槓桿的完整語義（W37 預告）：
- **借債本身不是罪**——在驗證假設之前精雕細琢是浪費（W19：MVP 就該借債！先確認有人要，再還）
- 災難來自**只借不還、甚至不記帳**：利息複利滾動，某天「加一個按鈕要三週」、每次上線都出事、資深工程師陸續離職（沒人想住在債務堆裡）——**開發速度的崩潰是漸進的，然後突然的**

Martin Fowler 的**技術債四象限**把債分類（審慎/魯莽 × 蓄意/無意）：
- **審慎蓄意**（「先上線驗證，Q3 排重構」）：健康的槓桿
- **魯莽蓄意**（「沒時間管品質了，堆上去再說」）：高利貸
- **審慎無意**（「現在才懂當初該怎麼設計」）：學習的自然產物，無可避免
- **魯莽無意**（「什麼是分層架構？」）：能力問題，不是取捨問題——這不叫借債，叫不知道自己在簽本票

### 1.4 三個管理機制（把債務從隱形變顯形）

1. **債務登記簿（debt register）**：每筆蓄意借的債記下來——位置、利息症狀（它拖慢了什麼）、估計還款成本——像財務的應付帳款一樣管理。沒有登記簿的技術債=沒有帳本的公司
2. **固定重構預算**：sprint 產能的 **15–20%** 固定還債，不與新功能搶排期（一旦要搶，新功能永遠贏——W23 探索/利用困境的微觀版：還債是「利用單位裡的探索」，要用預算池保護）
3. **ADR（架構決策紀錄）**：每個重大技術決策留一頁——情境、選項、取捨、決定——讓三年後的人知道「為什麼當年這樣做」。ADR 是對抗「審慎無意債」的疫苗：決策的 context 不隨人員流動蒸發

### 1.5 CTO 的翻譯課（本週的靈魂）

把工程語言翻成董事會語言：

| 工程說法 | 董事會翻譯 |
|---|---|
| 「要重構」 | 「償還債務本金，把**每個未來功能的邊際成本**降回正常水位」（W11 的語言） |
| 「技術債很重」 | 「我們的開發速度正在付複利利息——同樣的人力，產出逐季下滑 X%」 |
| 「先湊合上線」 | 「借一筆債換取三個月的市場驗證時間——還款計畫在 Q3，登記簿見」 |
| 「這個系統該重寫了」 | 「利息已超過重建成本——繼續付租不如買斷」（要求對方用數據證明） |

反向翻譯也一樣重要：董事會的「為什麼工程越來越慢」→ 打開債務登記簿看利息清單，而不是加人（Brooks 定律：對延遲的專案加人，只會更延遲——新人要人帶、溝通成本平方成長）。

### 📄 必讀一：Ward Cunningham,〈The WyCash Portfolio Management System〉(OOPSLA 1992)
🔗 http://c2.com/doc/oopsla92.html （免費，一頁經驗報告）

**讀法（約 15 分）：** 技術債的出生證明，全文極短。找到那段原始比喻（"Shipping first time code is like going into debt..."），注意 Cunningham 的原意常被遺忘的一半：**借債是為了加速學習，前提是「隨後用學到的東西重寫」**——他從沒說過債可以不還。

### 📄 必讀二：Martin Fowler,〈Technical Debt Quadrant〉
🔗 https://martinfowler.com/bliki/TechnicalDebtQuadrant.html （免費）

**讀法（約 15 分）：** 四象限的原典短文。順藤摸瓜可加讀同站〈Technical Debt〉條目。**帶著問題讀：** Fowler 為什麼堅持「魯莽無意」的爛 code 不算 debt 的正當用法？（提示：比喻的價值在「蓄意的取捨」——把爛攤子都叫債，比喻就失去管理意義。）

## 4. Quiz

**Q1（單選）** 一家電商評估自建薪資系統（工程團隊說「兩個月就能做完，比年費便宜」）。最有力的反對理由是：
- A. 兩個月的估計太樂觀
- B. 薪資系統是 context：自建的維運長尾＋工程資源的機會成本遠超授權費，且做到世界級顧客也無感
- C. 市售系統的功能比較多
- D. 自建系統不符合資安法規

**Q2（單選）** 依 Fowler 四象限，「我們知道這樣寫不乾淨，但要趕在展會前驗證需求，已排定下季重構」屬於：
- A. 魯莽蓄意
- B. 審慎蓄意——健康的槓桿使用
- C. 審慎無意
- D. 魯莽無意

**Q3（單選）** 技術債的「利息」具體是指：
- A. 銀行貸款的利息費用
- B. 每次開發都因繞過湊合方案而變慢——功能交付的邊際成本持續上升
- C. 伺服器的租用費用
- D. 工程師的加班費

**Q4（簡答）** 「供應商鎖定（lock-in）」如何從 W15 五力的角度傷害你？舉一個常見例子，並提出兩個在**簽約前**就能佈局的減損手段。

**Q5（簡答）** 董事會問：「工程團隊人數翻倍，功能產出反而變慢，為什麼？」——用技術債（利息）＋Brooks 定律組織一個兩分鐘的 CTO 級回答，並附上你的處方。

<details>
<summary>👉 展開解答與解說</summary>

**Q1：B。** 兩關全掛：core/context 關——薪資系統做得再好，顧客不會因此多下一張訂單；TCO 關——「兩個月做完」只是頭期款，法規更新、年度調整、資安修補的維運長尾要養一輩子，而這批工程師本可去做推薦引擎（機會成本）。「比年費便宜」是只算頭期款的典型錯帳。

**Q2：B。** 三要件俱全：知道取捨（審慎）、主動選擇（蓄意）、有還款計畫（排定重構）。少了第三件就滑向高利貸——「下季重構」若連續三季被新功能擠掉，就是登記簿與重構預算池（1.4）要接手的時刻。

**Q3：B。** 利息的載體是**時間**：同樣一個功能，在乾淨的程式碼基礎上三天、在債務堆上三週——差額就是利息，而且隨債務累積複利成長。這也是為什麼「利息>重建成本」時重寫才成立——要用數據（功能交付週期的趨勢）證明，不是憑感覺。

**Q4 參考答案：** 五力視角：lock-in 把你變成「轉換成本極高的買方」——供應商議價力（W15 ④）飆升：續約漲價、綁售模組、支援降級你都得吞（走不掉）。常見例：資料存在 SaaS 裡且無標準匯出格式、深度客製把流程焊死在單一 ERP。簽約前佈局：①**資料主權條款**——合約明定資料格式、完整匯出權與協助遷移義務；②**架構隔離**——用中間層/標準介面（API 抽象層）包住供應商，讓核心流程不直接依賴其私有格式；（加碼：談判時保留第二供應商的活口——W17 PARTS 的 Players 槓桿。）

**Q5 參考答案（示例）：**「兩個原因疊加。第一，我們在付**技術債的複利利息**：過去兩年為搶時程借的湊合方案沒有還款，現在每個新功能都要繞過它們——同一個功能的開發成本是兩年前的三倍，登記簿上前五大債務的利息清單在這裡（展示）。第二，**Brooks 定律**：對慢的專案加人，短期更慢——新人需要資深者帶（瓶頸資源被抽走，W27）、溝通路徑隨人數平方成長。所以人數翻倍沒有翻倍產出，反而攤薄了瓶頸。處方：①凍結非必要新功能一個季度，20% 產能固定還前五大債（預算池保護）；②停止加人，先把瓶頸（資深工程師的 review 產能）榨乾與擴容（W27 的二、四步順序）；③建立 ADR 與登記簿制度，讓債務從此顯形——三個月後用「功能交付週期」的趨勢向各位驗收。」
